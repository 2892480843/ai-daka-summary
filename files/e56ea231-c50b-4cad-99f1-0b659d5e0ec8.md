# 船山问道 · 迭代记录

> 数字人项目从「提示词建议」到「单文件 RAG 应用」的逐轮演进记录。

---

## 读这份记录前必看：关于「效果」的诚实说明

这个项目至今**没有跑过任何系统性测试**——没有测试题集、没有不同模型的对照、没有检索命中率的统计。因此本记录里「效果」一栏不会出现「实测变好」这种说法。每条改动的效果，归入以下四类之一，请按标签理解：

- **✓ 确认改善**：结构性、能力性的提升，无需测试即可验证（如「现在是独立文件」「现在支持智谱」）。
- **~ 意图改善·未实测**：设计上应该更好，但是否真的更好，没有验证过（如「RAG 应让回答扎根原典」）。
- **△ 代价 / 可能变差**：这次改动带来的副作用或退步风险。
- **✗ 已知弱点 / 未解决**：当时就清楚没做好、留到后面的问题。

把这四类分开，是为了让你知道：哪些是踏实的进展，哪些只是「方向对、未证实」，哪些是欠下的账。

## 第 0 轮 · 提示词架构 + 平台内演示

**改了什么**
提出船山先生的「五层人格架构」——本体论（气一元论 / 道器统一论）、认识论（行先知后）、历史哲学（理势合一）、人性论（性日生日成）、处境情感（遗民身份）。并用平台内 API 做了一个可对话的内嵌演示。

**为什么改**
针对历史人物数字人最常见的三种失败：变成「语录机」（只复述原文）、「概念漂移」（用词是船山的、内涵却是程朱或现代人的）、「遗民身份缺位」（少了血气与悲愤，人格是空的）。把这三点反过来，就是五层架构要守住的。

**效果**
- ✓ 确认改善：建立了人格骨架，这是后续所有版本的地基。
- △ 代价：它是「演示」不是「工具」——依赖平台内 API，不能离线、不能自己保存、用户拿不走。

---

## 第 1 轮 · 独立 HTML 单文件

**改了什么**
做成一个可在浏览器直接打开的单文件 `.html`：接 Anthropic API、流式输出（打字机效果）、深色书斋背景、对话可导出为 Markdown。

**为什么改**
你明确要「能在浏览器打开就能用」。核心是把演示升级成用户真正持有、可独立运行的工具。

**效果**
- ✓ 确认改善：从「演示」变「工具」——这是质变，可独立运行、可保存、可分享。
- △ 代价：只支持 Anthropic 一家，且需境外网络，国内用户门槛高。
- △ 可能变差：深色背景与「明代山水 / 古典书斋」的气质并不搭，偏科技感——这一点下一轮就被你推翻了，说明当时的视觉判断不对。

---

## 第 2 轮 · 多家 API + 山水背景

**改了什么**
两件事：(1) 从单一 Anthropic 扩展到 6 家提供商——智谱清言、Anthropic、DeepSeek、OpenAI、Moonshot、自定义端点，用 OpenAI 兼容格式统一调用；(2) 深色背景换成 SVG 手绘的明代山水（远山分层、云雾、松林、小亭），改用暖纸色调。

**为什么改**
你要求「支持智谱等多家 API」并「背景换明代山水」。多家 API 直接降低国内访问门槛（智谱有免费额度、可直连）；山水背景贴合主题。

**效果**
- ✓ 确认改善（可用性）：智谱 / DeepSeek 国内直连 + 免费额度，门槛大幅下降。这是整个项目里「能不能用起来」最关键的一步。
- ~ 意图改善（视觉）：山水比深色更贴题——但 SVG 是「程序画的山水」，意境有限，只到占位级，算不上真画意（下一轮被真迹替换）。
- ✗ 已知弱点：OpenAI 浏览器直连可能被 CORS 拦截。已在界面提示「遇阻请改用智谱 / DeepSeek」，属于提示规避，不是根治。

## 第 3 轮 · 限定原典（RAG 检索）

**改了什么**
这是最大的一轮结构改动。从你上传的 EPUB《船山遗书》抽取 12 种著作正文（思问录、周易外传、张子正蒙注、黄书、噩梦、尚书引义、读通鉴论、宋论、搔首问等），切成 188 个段落、内嵌进 HTML；每次提问先做检索、取出相关段落注入提示词；提示词加「最高规则」——只许用提供的原典作答，查不到就说「原典无载，不敢妄言」，禁止泛化、禁止用一般知识或网络信息。

**为什么改**
你要求「知识来源只能是《船山遗书》，不许泛化，不接受网络整合」。这把数字人从「扮演口吻」升级为「有据可依」，目标是防幻觉、防泛化。

**效果**（这一轮必须把好坏拆开讲，因为它「方向对、但实现粗」）
- ✓ 确认改善：原典确实被接进了回路，提示词里有了真实的书证，「来源限定」这个承诺在结构上成立了。
- ~ 意图改善·未实测：**是否真的检索到对的段落，从未验证过。**
- ✗ 已知弱点（检索方式）：用的是字面匹配（n-gram），比的是「字有没有重叠」而非「意思像不像」。你问「人性会不会变」、原文写「性日生而日成」，意思对应但字面几乎不重叠，**最该调出来的段落很可能被漏掉**。这是后续评审认定的头号待改项。
- ✗ 已知弱点（覆盖）：只内嵌 12 种，《读通鉴论》30 卷只放了 2 卷，大量史论问题会撞「无载」。
- ✗ 已知弱点（门控软）：「查不到说无载」只是一句嘱咐，模型在扮演压力下仍可能硬编，程序没有强制拦截。
- △ 代价：文件从约 30KB 膨胀到约 280KB（可接受）。

## 第 4 轮 · 三图背景 + 逐句白话

**改了什么**
两件事：(1) 把你上传的三张明代山水立轴压缩后内嵌，排成「三联屏」——图与图之间留间隔、上覆淡纸色蒙版保证文字可读；(2) 提示词加「逐句白话对照」最高规则：每写一句文言，紧接括号附该句白话，禁止只在段末统一翻译、禁止跳句。

**为什么改**
你上传了真迹并要求「背景由三张图构成、留间隔」「回答每句附白话」。真迹替代手绘；逐句白话对应你「古文必附白话」的固定偏好。

**效果**
- ✓ 确认改善（背景）：三张真迹的意境远胜上一轮的 SVG 手绘，这是确定的视觉提升。
- ~ 意图满足·未实测（白话）：格式要求写清楚了、给了范例锚定，但模型是否真的「每句不漏」没验证。
- △ 代价 / 可能变差（约束打架）：现在同时压着三条强约束——只用原典 + 人格鲜活 + 逐句白话。三者会互相拉扯，**小模型尤其可能为了凑白话翻译把话说僵、丢掉思想家的气血**，也可能偷懒只在段末翻译。约束越多，小模型越容易顾此失彼。
- △ 代价：文件涨到约 1.1MB（三图内嵌所致，仍属单文件可接受范围）。

## 第 5 轮 · 评审（无代码改动）

**改了什么**
没动代码。你问「对比成熟的历史人物数字人，最该优化什么」，我输出了一份诊断。

**为什么**
在继续加功能前，先盘清楚欠了哪些账、哪一项改造性价比最高。

**结论（按优先级）**
1. 检索层：字面匹配 → 语义检索（向量 / embedding），这是花同样力气收益最大的一项。
2. 多轮追问会断根：现在只拿「最新一句」去检索，你回「继续 / 展开一下」时检索不到东西，那一轮就悬空。
3. 看不到原文：模型说「原典如此写道」，但你点不开、无法当场核对它有没有改写或编造。
4. 三重约束打架（见第 4 轮）。
5. 幻觉门控太软：应在检索分数过低时由程序强制「只能答无载」，而非靠模型自觉。
6. 没有评测：所有「立场会到位」的断言都没用测试题验证过。

**效果**
- 不适用——这是诊断不是改动。但它的价值是把「下一步该往哪走」定下来了：评审之前我们一直在加功能，评审之后才有了优先级。

## 一句话总结这几轮的轨迹

前几轮是**把工具做出来、做能用、做好看**——这部分进展是踏实的、可确认的。后两轮触及**「回答有没有真的扎根原典」**这个更难的内核，方向对，但实现还粗，而且至今没有任何测试来告诉我们它到底做到了几分。

下一步若要继续，**最该先补的不是再加功能，而是先搭一组测试题**——否则后面每一项优化（语义检索、门控、追问修复）都还是在「应该会更好」的层面打转，无法确认。
